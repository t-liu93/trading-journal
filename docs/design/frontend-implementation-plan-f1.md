# Frontend Phase F1 — Account CRUD UI

**Language:** English | [中文](./frontend-implementation-plan-f1.zh.md)

> Status: **DRAFT v0.1** (2026-05-19). Companion to [frontend-implementation-plan.md](./frontend-implementation-plan.md) (F0 base scaffold) and [data-model.md](./data-model.md). Iterate here before writing code.

## 1. Purpose

F0 proved the auth + router + cookie wiring works end-to-end. F1 puts the first real **data-bearing screen** on top of that scaffold: full Account CRUD UI on top of the Phase 4 backend endpoints.

Beyond the immediate user value, F1 establishes the **reusable patterns** for every future entity (Position, Trade, TradePlan, …):

- API type generation from `/openapi.json` (no more hand-written interfaces)
- Resource API client module pattern (`src/api/<resource>.ts`)
- View + composable separation (`AccountsView` + `useAccounts`)
- Modal-based create/edit form (shared between the two flows)
- Confirm-dialog for destructive actions
- Skeleton loading + empty state + toast feedback
- Soft-delete with archived filter

Get these right once and F2 (Position) is mostly applying the same recipe.

## 2. Scope

### In scope (this plan)

- **`openapi-typescript` codegen** wired up; `src/api/schema.d.ts` checked in; existing hand-written `User` type replaced
- **`src/api/accounts.ts`** — typed wrapper around the 5 backend endpoints
- **`useAccounts()` composable** — page-local state + actions
- **`AuthenticatedLayout.vue`** component — header + nav + content slot; shared by Dashboard and Accounts
- **`AccountsView.vue`** — `n-data-table` list + skeleton + empty state + "Show archived" toggle + "+ New account" button
- **`AccountFormModal.vue`** — shared modal handling both create and edit modes, with proper Naive UI form validation
- **Delete (archive) confirmation** via `useDialog().warning()`
- **Updated `DashboardView`** — small "Your accounts" card linking to `/accounts`; placeholder cards for future entities so the roadmap is visible in-app
- **README updates** — codegen workflow documented
- **Backend regression** — Phase 2–4 pytest stays at 46/46

### Explicitly NOT in scope (deferred)

- **Pinia store for accounts** — F1 keeps state page-local in a composable; promote to a store only when ≥2 components need shared reactivity
- **Bulk operations** (multi-select delete / archive)
- **Search / filter** beyond the archived toggle
- **Sort customisation** — default `created_at` order, no header click sorting
- **Pagination** — realistically ≤10 accounts per user; revisit when needed
- **Account detail page** (separate route showing positions belonging to the account) — that's F2 territory
- **Hard-delete option** — only soft-delete in MVP
- **Frontend unit tests** (Vitest) — defer until ≥1 non-trivial pure function exists
- **`openapi-fetch` typed client** — `openapi-typescript` types + our existing `http.ts` are enough
- **Tailwind / custom theme** — Naive UI defaults still fit

## 5. Phased plan

Sequential sub-phases. Each has **Goal**, **Tasks**, **Manual verification**, **Acceptance**. Don't start N+1 until N's acceptance passes.

### F1.1 — `openapi-typescript` codegen

**Goal.** A single source of truth for API types: backend Pydantic → `/openapi.json` → `schema.d.ts` → import in TS. No more hand-maintained `User` interface.

**Tasks.**
1. `npm install -D openapi-typescript`
2. Add `package.json` scripts:
   ```json
   {
     "scripts": {
       "codegen": "openapi-typescript http://127.0.0.1:8000/openapi.json -o src/api/schema.d.ts"
     }
   }
   ```
3. With uvicorn running, run `npm run codegen` once to produce `src/api/schema.d.ts`
4. **Replace** the hand-written `User` interface in `src/api/types.ts` with:
   ```ts
   import type { components } from './schema'
   export type User = components['schemas']['UserRead']
   ```
5. Keep in `src/api/types.ts`: `ApiError` class, `ApiErrorDetail`, `RegisterPayload`, `LoginPayload` (these are app-shaped, not API-shaped)
6. Document the codegen workflow in `frontend/README.md`:
   - When to regenerate (after any backend schema change)
   - That `schema.d.ts` is committed to git
   - That backend must be running locally to regenerate

**Manual verification.**
```bash
cd frontend
npm install
# Make sure backend is up on :8000, then:
npm run codegen
# → writes src/api/schema.d.ts; git diff shows the changes
npm run build
# → vue-tsc + vite both pass with the new generated types
```

**Acceptance.** `src/api/schema.d.ts` exists and is checked in; no manually-written model interface remains anywhere; `npm run build` passes.

### F1.2 — Account API client

**Goal.** A typed function per backend endpoint, isolated from view code.

**Tasks.**

1. Create `src/api/accounts.ts`:
   ```ts
   import type { components } from './schema'
   import { http } from './http'

   export type Account       = components['schemas']['AccountRead']
   export type AccountCreate = components['schemas']['AccountCreate']
   export type AccountUpdate = components['schemas']['AccountUpdate']

   export const accountsApi = {
     list:   (includeArchived = false) => http.get(`/accounts?include_archived=${includeArchived}`) as Promise<Account[]>,
     get:    (id: string)              => http.get(`/accounts/${id}`)                              as Promise<Account>,
     create: (payload: AccountCreate)  => http.post('/accounts', payload)                          as Promise<Account>,
     update: (id: string, payload: AccountUpdate) => http.patch(`/accounts/${id}`, payload)         as Promise<Account>,
     remove: (id: string)              => http.delete(`/accounts/${id}`)                           as Promise<null>,
   }
   ```
2. (No view consumer yet — that's F1.3+.)

**Manual verification.** None standalone — exercised in F1.3.

**Acceptance.** File compiles; type signatures match backend.

### F1.3 — `AuthenticatedLayout` + `useAccounts` composable + `AccountsView` shell

**Goal.** Shared layout extracted. `/accounts` route renders an empty table with skeleton + empty state. No create/edit/delete yet.

**Tasks.**

1. **`components/AuthenticatedLayout.vue`**:
   - `<n-layout>` with `<n-layout-header bordered>`
   - Header: app title (left), nav (`Dashboard | Accounts`), user email + Logout button (right)
   - Logout reuses the same pattern from current `DashboardView`
   - Content via `<slot />`
2. **`composables/useAccounts.ts`**:
   ```ts
   export function useAccounts() {
     const accounts        = ref<Account[]>([])
     const loading         = ref(false)
     const error           = ref<string | null>(null)
     const includeArchived = ref(false)

     async function refresh() {
       loading.value = true
       error.value = null
       try {
         accounts.value = await accountsApi.list(includeArchived.value)
       } catch (err) {
         error.value = err instanceof ApiError ? err.message : 'Failed to load accounts.'
       } finally {
         loading.value = false
       }
     }

     // Auto-refresh when toggle flips
     watch(includeArchived, refresh)

     return { accounts, loading, error, includeArchived, refresh }
   }
   ```
3. **`views/AccountsView.vue`** (first cut, no modal yet):
   - Wraps `<AuthenticatedLayout>`
   - Page header with title + right-aligned `<n-switch>` for "Show archived" + `<n-button>` "+ New account" (disabled in this sub-phase)
   - `<n-data-table>` columns: Name, Broker, Type, Currency, Notes, Created, Actions (placeholder)
   - Loading: `<n-data-table :loading="loading">` (built-in skeleton)
   - Empty state: `<n-empty>` "No accounts yet — create one to get started" with a `<n-button>` that's also disabled in this sub-phase
   - Archived rows: styled with `depth="3"` (gray) + a small `<n-tag size="small">archived</n-tag>` in the Name column
4. **`router/index.ts`** add:
   ```ts
   {
     path: '/accounts',
     name: 'accounts',
     component: () => import('../views/AccountsView.vue'),
     meta: { requiresAuth: true },
   }
   ```
5. **`views/DashboardView.vue`**: refactor to use `<AuthenticatedLayout>`. Replace the "coming in F1" placeholder with a `<n-card>` "Your accounts" + a `<RouterLink to="/accounts">` "Manage accounts →". Add two more placeholder cards: "Positions (F2)" and "Trades (F3)" so the upcoming roadmap is visible in-app.

**Manual verification.**

- Log in, navigate to `/accounts` via the new header nav
- Empty DB → see `<n-empty>` "No accounts yet"
- Add an account via curl (or wait for F1.4); refresh → row appears
- Toggle "Show archived" — table re-fetches (Network tab shows `GET /accounts?include_archived=true`)
- `/` still shows the Dashboard with the new "Your accounts" card linking to `/accounts`

**Acceptance.** All of the above; build green; no console errors.

### F1.4 — `AccountFormModal` (create + edit)

**Goal.** Click "+ New account" or row-edit → modal with form → submit → list refreshes.

**Tasks.**

1. **`components/AccountFormModal.vue`** props:
   - `:show` (boolean, two-way via `v-model:show`)
   - `:mode` — `'create' | 'edit'`
   - `:initial` — partial `Account` for edit mode (ignored in create mode)
   - Emit: `@saved` with the resulting `Account` (parent decides what to do — typically `refresh()` + close)
2. Form fields (all Naive UI):
   - `name` — `<n-input>`; rule: required, max 255
   - `broker` — `<n-input>`; rule: required, max 255
   - `account_type` — `<n-select>` with options `cash | margin | paper`; rule: required
   - `base_currency` — `<n-select>` with common options (USD, EUR, GBP, JPY, CHF, CAD, AUD, HKD) **plus** `filterable` for free-typing rarer codes; rule: required + regex `^[A-Z]{3}$`
   - `notes` — `<n-input type="textarea">`; optional
3. Submit handler: build payload (dropping fields the user left at default), call `accountsApi.create` or `accountsApi.update`, on success `useMessage().success(...)` + `emit('saved', result)` + close modal
4. Error handler: `useMessage().error(err.message)` if `ApiError`; re-throw otherwise
5. Validation timing: validate on submit only (don't nag while typing) — Naive UI's `formRef.value?.validate()` returns a promise
6. Wire in `AccountsView`:
   - "+ New account" button → opens modal in create mode
   - Row Actions column → "Edit" button (only on non-archived rows) → opens modal in edit mode with `:initial="row"`
   - On `@saved` → call `refresh()`, close modal

**Manual verification.**

- Click "+ New account" → modal opens, all fields empty
- Submit with no values → field-level red errors appear
- Fill in valid values → modal closes, success toast, row appears in table
- Click "Edit" on a row → modal opens with pre-filled values, currency dropdown shows the existing value
- Change `notes`, save → row updates, success toast
- Try `base_currency` = `usd` → validation error before any request fires
- Submit with backend-invalid value (e.g., add `?` somehow — typically not possible through the form, but try via DevTools) → ApiError surfaced via toast

**Acceptance.** Create works, edit works, validation triggers on submit, success/error toasts surface, list auto-refreshes.

### F1.5 — Delete (archive) confirmation flow

**Goal.** Row "Archive" action shows a confirmation dialog; on confirm, soft-deletes via API, refreshes, toasts.

**Tasks.**

1. In `AccountsView`'s Actions column, add an "Archive" button next to "Edit" (only on non-archived rows)
2. Click handler:
   ```ts
   const dialog = useDialog()

   function handleArchive(row: Account) {
     dialog.warning({
       title: 'Archive this account?',
       content: `"${row.name}" will be hidden from your default list. You can recover it any time by toggling "Show archived".`,
       positiveText: 'Archive',
       negativeText: 'Cancel',
       onPositiveClick: async () => {
         try {
           await accountsApi.remove(row.id)
           message.success(`Archived "${row.name}"`)
           await refresh()
         } catch (err) {
           if (err instanceof ApiError) message.error(err.message)
           else throw err
         }
       },
     })
   }
   ```
3. Word the dialog as "Archive" not "Delete" — emphasises recoverability

**Manual verification.**

- Click "Archive" on a row → dialog shows account name + recoverability hint
- Click "Cancel" → nothing happens
- Click "Archive" (positive) → row disappears from list, toast "Archived X"
- Toggle "Show archived" → archived row reappears, dimmed, with `[archived]` tag, no Edit/Archive buttons

**Acceptance.** Full flow works; archived state is reversible only via the toggle (no UI to unarchive in F1 — see §9).

## 6. Codegen workflow

Operational details for the `openapi-typescript` setup introduced in F1.1.

### When to regenerate

- After **any backend change** that affects request/response shapes (new endpoint, new field, renamed field, type change, enum value added)
- After a backend dependency bump that changes Pydantic v2 → vX behaviour (rare)

### How to regenerate

```bash
# Backend must be running on :8000
cd frontend
npm run codegen
git diff src/api/schema.d.ts   # review what changed
npm run build                  # confirm no breakage downstream
```

### What to commit

- `src/api/schema.d.ts` is **committed**, treated as generated-but-tracked (so freshly cloned repos compile without first running the backend)
- Don't hand-edit it — re-run `codegen` instead

### Optional follow-ups (post-F1)

- **CI check**: a job that re-runs codegen and `git diff --exit-code` to catch "backend changed but forgot to regenerate"
- **pre-commit hook**: same idea, locally enforced
